# 产品需求文档 (PRD)

## 1. 产品背景与目标
当前多智能体(Multi-Agent)协作主要依赖松散的聊天流与文件交换(如手动维护 `BLOCK.md`, `KANBAN.json`)，极易导致状态丢失、上下文碎片化和调度灾难。
**io-manager** 的核心目标是：提供一个标准化的 **AI-PM 独立后端系统**。将“人类老板的宏观需求”转换为“AI打工人的微观执行态”，同时提供上帝视角的大盘追踪，彻底自动化整个 AI 开发协作生命周期。

## 2. 核心用户画像与痛点
- **分配者/老板 (如: 用户博)**：给出高阶需求，只想看进度，不想管理琐碎的 AI 拆解过程。**痛点**：缺乏实时透视进度的大盘，必须去问 Agent 才知道进展。
- **管理智能体 (AI-PM/如: 刘亦菲)**：负责接收老板需求，拆解并调度下属干活。**痛点**：缺乏维护任务状态机的平台支撑，状态机流转易死锁。
- **执行智能体 (Worker Agents/如: 研发、测试)**：负责执行具体微观任务。**痛点**：需要清晰的输入输出契约与前置后置依赖卡口。

## 3. 核心功能模块 (Features)

### 3.1 蓝图解析引擎 (Project & Blueprint Engine)
- **需求接收与拆解**：接收非结构化需求文本，生成结构化蓝图草案(Project Blueprint)。
- **蓝图节点管理**：增删改查蓝图节点（里程碑 -> 模块 -> 任务）。支持版本控制。

### 3.2 动态看板与工作流引擎 (Kanban & Workflow)
- **状态机流转**：定义标准状态 `Triage(分发中) -> Backlog(待办) -> InProgress(进行中) -> Blocked(阻塞) -> Review(验收) -> Done(完成)`。
- **依赖锁机制**：任务 B 依赖任务 A 时，自动挂起任务 B。
- **超时与接管告警**：处于 `InProgress` 或 `Blocked` 状态超时的任务，自动向 AI-PM 抛出接管事件。

### 3.3 智能体协作中枢 (Agent Hub & Webhook)
- **任务领用派发机制 (Pub/Sub)**：基于不同打工人的标签(Java开发、前端、测试)精准派发指令。
- **AI 打工人上报接口**：提供标准 API 供执行节点提交流转节点报告与制品 (Artifacts)。

### 3.4 老板大盘与可视化 (Boss Dashboard - Module 4)
- **项目燃尽图与进度树**：以树状图或时间轴呈现整体拆解进度与各个 Agent 的忙碌/空闲状态。
- **阻塞告警透出**：将深层技术卡点向上抽象成人类可读的风险播报。

## 4. 全局非功能性约束
- **超高并发I/O**：因涉及大量 AI 节点 HTTP 等待回调，必须支撑大吞吐挂起连接。
- **不可篡改审计**：底层任务图节点的变更记录需完整保留 (Event Sourcing 思想)，方便朔源。
